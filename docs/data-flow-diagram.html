Study Crossing — Data Flow Diagram (Current Build)

Rebuilt DFD for the live SQLite system · Yourdon–DeMarco notation · compared against the original Software-Engineering-Report plan (May 2026)

0 · Why this DFD was redrawn Context

The original software-engineering-report.html contained a DFD, but it described a system that no longer matches the code. This document replaces it with a DFD of the system as actually built, then explains every difference and the reason for it (§3 in prose, §5 as a table).

The original report stated, word for word: “The app runs entirely on-device with no network layer, no database, and no server… There is no login system and no admin dashboard in the current design… The DFD section is hypothetical: it assumes the mock data layer is replaced by REST endpoints and a remote database.”

In other words, the planned DFD modelled one Player talking to a hypothetical remote backend that did not exist yet. The shipped app is different on both counts: it has a real, local SQLite database (StudyCrossing.db, 7 tables) and three roles (Student, Teacher, Admin) behind a login. The diagrams below therefore document real data flows, not a future hypothetical.

3 · Detailed Explanation Narrative

3.1 · Reading the current DFD

At the context level (Figure 1) the whole application is treated as a single process — process 0, “Study Crossing System” — surrounded by the three people who use it: the Student, the Teacher and the Admin. Every arrow that crosses the system boundary is a packet of data. The Student sends in credentials, submitted answers, lesson notes and profile edits, and receives quests, grades, bell payouts and history back. The Teacher sends in lesson uploads and class setup and receives lesson lists and class-progress reports. The Admin sends in account and lesson edits and receives the management lists and counts. The context diagram deliberately hides all internal storage; it only answers the question “who talks to the system, and what data passes between them?”.

The Level-1 diagram then lifts the lid on process 0 and shows the nine internal processes and the eight data stores they read and write. Because nine processes in one frame would be unreadable, the decomposition is drawn as two partitions that follow the natural split in the app: Figure 2a covers everything a Student does while learning, and Figure 2b covers everything a Teacher or Admin does while managing the system. Process 1.0 (Authentication) is the one process common to all three roles, so it appears at the top of both partitions.

In the learning partition (Figure 2a) the flow is a clean pipeline. Process 1.0 checks the student’s credentials against the students store (D1) — or inserts a new row when registering. Once in, process 2.0 Quest Delivery reads a quest definition from the lessons store (D4) and the student’s completion flags from progress (D5), and hands the assembled lesson + game configuration back to the student. When the student answers, process 3.0 Answer Validation fetches the correct key independently from D4 — it never trusts the answer the client claims is right — and returns a correct/wrong verdict plus the remaining apples. On a win or a final miss it passes a quest-outcome packet to process 4.0 Progress & Reward, which is the only writer of results: it appends a row to sessions (D6), sets the done flag in progress (D5), adds the earned bells to the student’s balance in students (D1), and returns the grade. Separately, process 5.0 Notes Management reads and writes the student’s private annotations in notes (D7). The shapes carry the meaning: circles are work the system does, the paired lines are data sitting at rest, and the labelled arrows are data in motion between them.

In the management partition (Figure 2b) the same process 1.0 now validates Teacher and Admin logins against the staff store (D2). The Admin drives three CRUD processes: 6.0 Student Account Management writes the students store (D1) and assigns a class value into classes (D3); 7.0 Staff Management inserts and deletes teacher accounts in staff (D2); and 8.0 Lesson Management maintains the lessons store (D4). Process 8.0 is shared — a Teacher reaches the very same process to upload and edit lessons, which is why two entities point at it. Finally, 9.0 Class & Progress Monitoring lets the Teacher create classes and assign students in classes (D3), and — crucially — it reads three stores at once (students, progress, sessions) to compose the read-only class-progress report it sends back to the Teacher. That single process is where the learning side and the management side of the data finally meet.

Notice that D4 (lessons) appears in both partitions: the Admin/Teacher write it through 8.0 in Figure 2b, and the Student reads it through 2.0 / 3.0 in Figure 2a. That shared store is the whole point of the redesign — the same table staff edit is the table the gameplay is built from. All seven stores are real database tables, so they map one-to-one onto the seven ERD entities.

3.2 · How this differs from the previous DFD

The most fundamental difference is what the diagram is about. The DFD in the original Software Engineering Report was explicitly hypothetical: the app of that era had no database at all, so the report drew an imagined future in which a remote REST backend would one day serve a single player. The diagram on this page is the opposite — it documents the system that was actually built and shipped, backed by a real, on-device SQLite database. The earlier DFD described software that did not exist yet; this one describes software that does.

From that one change everything else follows. The original context diagram had a single external entity called “Player”, because there was only ever one kind of user. The current context diagram has three — Student, Teacher and Admin — because the app grew a content-authoring side (teachers upload lessons and watch class progress) and an administration side (the admin manages every account). A single-player model simply had nowhere to put those responsibilities, so new entities had to appear.

The process count rose from three to nine for the same reason: each new capability is a new data transformation. The original three processes — quest delivery, answer validation, and progress recording — all survive, almost unchanged, as processes 2.0, 3.0 and 4.0. Around them the build added six more: authentication (1.0), notes management (5.0), student account management (6.0), staff management (7.0), lesson management (8.0) and class/progress monitoring (9.0). None of these existed in the planned diagram because none of those features existed in the planned app.

The data stores tell the rest of the story. The old DFD had only two: D1 “Quest Bank” and D2 “Session History”. The new DFD has seven — students, staff, classes, lessons, progress, sessions and notes — one per real SQLite table. Notably, the quest content ended up back inside the database, close to the original plan's “Quest Bank” intent: the lessons table is now the single source the gameplay quests are built from, so the same rows admins and teachers edit are the rows students play. (An earlier build had kept the quests as a hard-coded in-memory pool; folding them into the lessons table is what lets the content be edited without touching code.)

Two further differences are worth naming. First, authentication did not exist in the planned design at all — the old report says so in plain words — whereas it is now process 1.0 and the gatekeeper for the whole system; the student id it returns becomes the filter that scopes every later per-student read and write. Second, the notation itself changed: the original report drew its DFD in Gane & Sarson symbols (rounded-rectangle processes, open-ended rectangle stores), while this redraw uses the Yourdon–DeMarco standard (circular processes, paired-line stores). That last change is purely cosmetic — it does not alter any data flow — but it is the most immediately visible difference between the two documents.

In short, the diagram grew from a 1-entity, 3-process, 2-store sketch of an imagined backend into a 3-entity, 9-process, 7-store model of a working, role-based, database-backed application. The itemised, row-by-row version of this same comparison — with the “why” for each line — is given in the table in §5.

5 · Planned vs Built — what changed & why Comparison

Side-by-side of the original Software Engineering Report (May 2026) and the current build, with the reason for each change.

AspectOriginally plannedCurrent buildWhy it changed
Persistence No database. In-memory MockQuestRepository. DFD was an imagined future backend. Real local SQLite — StudyCrossing.db, 7 tables via DatabaseHelper + 2 facades. Data must survive restarts and be per-user (coins, progress, notes). The report itself named QuestRepository as the “swap point”; that seam was used to drop in a real store.
Backend style Hypothetical remote REST server + remote DB. On-device relational DB (SQLite); no network. It is a standalone classroom Android app — no server / hosting to run. SQLite gives real persistence, CRUD and table relationships locally, satisfying the requirement without infrastructure.
External entities 1 — Player. 3 — Student, Teacher, Admin. A single-player model cannot author content or supervise learners. Teachers were added to upload lessons & watch class progress; an Admin to manage all accounts.
Authentication None (“no login system”). Login + Register; role routing; validated against students / staff tables. Multiple roles need identity + access control, and per-student data needs a logged-in id (currentStudentId) to scope every query.
Data stores 2 — D1 Quest Bank, D2 Session History. 7 — students, staff, classes, lessons, progress, sessions, notes (all real tables). New entities appeared (staff, classes, per-student progress / sessions / notes), and the quest content folded into the lessons table instead of a separate store.
Quest data location Inside the DB (D1 Quest Bank). Inside the DB — the lessons table, built into quests at runtime. So admins/teachers can edit the actual gameplay content. A game_type column says which mini-game each row becomes; drawable/video resources are referenced by id, and quest_id in progress/notes is now a real reference to lessons.id.
Processes 3 — quest delivery, answer validation, progress. 9 — the original 3 + auth, notes, student mgmt, staff mgmt, lesson mgmt, class/progress monitoring. Every new role capability is a new data transform, so the DFD gained one process per capability.
Lesson content Hard-coded in the quest definitions. CRUD via the lessons table + device image upload. Content has to be editable by non-developers (teachers/admin), which is the whole point of an authoring screen.
Quests 8 quests. 14 quests. Content was expanded (Malaysia-school themed quests, wider subject coverage).
Screens (Activities) 12 Activities, 2 Fragments. 21 Activities, 2 Fragments. Added Login, Register, Admin (×6 screens), Teacher (×3 screens) and a student Lesson-Note screen.
Student notes Not in the plan. notes table + LessonNoteActivity. Added a personal study-notes feature so students can annotate lessons they have played.

The one-line story

The plan was a single-player, no-database app whose DFD merely imagined a future remote backend. The build turned that imagined backend into a real on-device SQLite database and grew the app from one user into a three-role system (students play; teachers author lessons & monitor classes; an admin manages all accounts). Every extra process and data store in the new DFD exists to serve one of those three roles — which is exactly why the original 1-entity / 3-process / 2-store diagram had to be redrawn as a 3-entity / 9-process / 7-store one.
