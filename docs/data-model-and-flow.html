This ERD shows only what actually exists in StudyCrossing.db. STUDENTS is the hub: one student owns many PROGRESS, SESSIONS and NOTES rows, each linked by the real student_id value (solid lines).
CLASSES → STUDENTS is dashed: a student's class is stored as the text form column (e.g. "Classroom A") and matched to classes.name by value — there is no enforced FK.
LESSONS is now the single source of the game content: the quests students play are built from this table (IslandDataStore.buildQuestPool() reads it), and the new game_type column decides which mini-game each row becomes (MCQ / slot / word-track / fossil). So quest_id in PROGRESS and NOTES is a real reference to lessons.id (solid lines) — finishing or annotating a quest points back at the lesson row it came from. fill_frame / show_lesson_image are display flags that preserve a couple of special cases (crop-fill images; the fossil intro that hides its phase-1 picture).
STAFF stays standalone: it only holds the Admin / Teacher login accounts (one admin seeded; teachers added / removed by the admin), with no link to a class or student.

7 · SQLite Connection Deep-Dive Database Internals

Every database connection in the project, traced end to end — from the single Application-level wiring, through the two facade objects, into DatabaseHelper, and onto the StudyCrossing.db file. All file/line references are to the real Kotlin source.

7.1 · One wiring point, two helper handles, one file

The whole database is wired up in exactly one place: MyApp.onCreate(), which Android runs before any Activity. It hands each facade a DatabaseHelper. Note there are two DatabaseHelper instances (one per facade) but they open the same physical file — SQLite serialises access, so two handles to one file is safe for this single-user app.

// MyApp.kt — runs first, builds both connections
class MyApp : Application() {
    override fun onCreate() {
        super.onCreate()
        AdminDataStore.init(this)   // → DatabaseHelper #1
        IslandDataStore.init(this)  // → DatabaseHelper #2 (same StudyCrossing.db)
    }
}

// each facade keeps its own handle, created in init()
object AdminDataStore {
    private lateinit var db: DatabaseHelper
    fun init(context: Context) { db = DatabaseHelper(context.applicationContext) }
}

7.2 · Lazy creation & destructive versioning

Constructing DatabaseHelper touches no disk. The .db file is created the first time readableDatabase / writableDatabase is read — i.e. the first login query. That first touch fires onCreate(), which runs 7 CREATE TABLE statements then 5 seed methods. DATABASE_VERSION = 4; bumping it fires onUpgrade(), which DROPs all 7 tables and calls onCreate() again — a clean destructive reset (no row migration), which is acceptable for a student demo.

Seed method (DatabaseHelper)TableRows inserted
seedStaffstaff2 — admin / teacher, both password 1234
seedStudentsstudents7 — ids 1..7 (AUTOINCREMENT), all password 1234
seedClassesclasses2 — Classroom A / Classroom B
seedLessonslessons14 — the full quest set (Math 5 / English 5 / GK 4)
seedProgressprogressper-student completion rows so teachers see varied progress

7.3 · The connection chain — UI never touches SQL

The architecture rule is one-directional: an Activity/Adapter calls a facade method → the facade forwards to a DatabaseHelper method → DatabaseHelper runs the raw SQL and returns plain Kotlin types (Int id, Long row id, ArrayList, String, Boolean). No SQL string ever appears in UI code, which is why swapping the old in-memory lists for SQLite changed zero Activities.

// one read travelling down the chain (student login)
LoginActivity        → AdminDataStore.loginStudent(u, p)        // UI layer
AdminDataStore       → db.loginStudent(u, p)                    // facade forwards
DatabaseHelper       → SELECT id FROM students                  // raw SQL
                       WHERE username=? AND password=?
returns              ← Int  (the student id, or -1 if no row)

There is one deliberate crossover: IslandDataStore also needs the students row (for coins, name, avatar). 7.5 · currentStudentId — the session scope

At login, IslandDataStore.setCurrentStudent(id) stores the logged-in id once. Every per-student call (getCompletedIds, insertSession, getNote, …) passes that id as the student_id filter — the application-level equivalent of a permanent WHERE student_id = ?. Logging out and back in as another student simply re-points this one field; no query changes.

7.7 · CRUD conventions (the project's SQLite fingerprint)

Every helper method follows the same hand-written pattern from the project rules — no Room, no ORM, no .use{}:

// 1) connection fetched LOCALLY each call (never cached as a field)
val db = writableDatabase

// 2) writes use ContentValues().apply { put(...) }
val values = ContentValues().apply {
    put(COLUMN_USERNAME, username)
    put(COLUMN_COINS, coins)
}
return db.insert(TABLE_STUDENT, null, values)  // returns Long row id; caller checks != -1L

// 3) reads are parameterised — ? placeholders + selectionArgs (no string concat)
val selection = "username = ? AND password = ?"
val selectionArgs = arrayOf(username, password)
val cursor = db.query(TABLE_STUDENT, null, selection, selectionArgs, null, null, null)

// 4) cursor closed MANUALLY right before return — no .use { } block
var id = -1
if (cursor.moveToFirst()) id = cursor.getInt(cursor.getColumnIndexOrThrow(COLUMN_ID))
cursor.close()
return id

7.8 · Multi-table write — solving one quest

A single correct answer fans out to three tables, all scoped to currentStudentId (the full call order is the sequence in §5):

RewardActivity.onCreate (score passed in)
  1. IslandDataStore.recordSession(subject, grade, coins)   // always runs
        → DatabaseHelper.insertSession   → INSERT INTO sessions(...)
        → AdminDataStore.addCoins(id, n) → DatabaseHelper.addCoins
              → SELECT coins → UPDATE students SET coins = old + n  // read-modify-write
  2. if (score >= 90) IslandDataStore.markDone(questId)        // only on S grade
        → DatabaseHelper.markDone        → INSERT INTO progress(student_id, quest_id)

addCoins is a read-modify-write (SELECT then UPDATE), not a single atomic SQL statement. For this single-user offline app that is fine; a concurrent multi-writer system would instead use UPDATE … SET coins = coins + ?.

7.9 · Soft links — no enforced foreign keys

None of the 7 CREATE TABLE statements declare a FOREIGN KEY clause. Every "relationship" in the ERD is enforced in Kotlin, by passing matching ids/strings — not by the database:

LinkHow it is matchedEnforced by
progress / sessions / notes .student_id → students .idreal per-row id passed as a query argKotlin (no SQL FK)
students .form ↔ classes .namematched by text value (e.g. "Classroom A")Kotlin (soft / dashed in ERD)
progress / notes .quest_id → lessons .idreal lesson row id passed as a query argKotlin (real table id, still no SQL FK constraint)

Because there are no FK constraints, deletes do not cascade. Wiping a student's data is done explicitly by resetAll() calling clearProgress + clearSessions + clearNotes in turn; deleteStudent removes only the students row and leaves orphan progress/session/note rows behind (harmless here, since every read is already filtered by the current student's id).

Phase 1 is the lesson (NPC + field-guide panel); Phase 2 is the interactive game that loads when the player taps Start Challenge. Each subject now mixes types — Math uses Slot Machine + Multiple Choice, English uses Word Track + Multiple Choice, and GK uses Fossil Excavation + Multiple Choice. A wrong answer costs one of three apples and the player keeps trying the same mini-game (its state is preserved — e.g. the slot reels are not reset); losing all three apples ends the quest at score 0, while a correct answer scores 95. RewardActivity turns the score into a letter grade and bells (95 → grade S, 0 → grade C; coins = score × 10).
